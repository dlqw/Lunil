# Lunil 0.10.0-alpha.1

## Lua language-version contract

This development release establishes the explicit Lua language-version identity used by the
compiler, canonical IR, runtime state, hosting API, workspace, and CLI.

- Lua 5.1 through Lua 5.5 are represented as distinct version identities.
- Lua 5.4 remains the default and the 0.9.x compatibility baseline.
- `--lua-version`, `luaVersion`, and `LUNIL_LUA_VERSION` select the requested contract.
- The selected version is included in canonical IR and JIT module identity data.
- A runtime state rejects modules from a different Lua language contract.
- Existing Lua 5.4 source, chunk, standard-library, interpreter, and JIT behavior remains the
  default and compatibility baseline.
- Selecting an as-yet unimplemented version fails explicitly instead of silently applying Lua 5.4
  semantics.

## Lua 5.3 compatibility slice

- Lua 5.3 source compilation and interpreter/JIT hosting are enabled with Lua 5.3 module identity.
- Lua 5.4-only local attributes are rejected by the Lua 5.3 parser.
- Lua 5.3 standard-library installation omits the Lua 5.4-only `warn` and `coroutine.close`
  surface.
- A separate Lua 5.3 binary-chunk reader, opcode model, and canonical converter import verified
  Lua 5.3 chunks without routing them through the Lua 5.4 format adapter.
- Lua 5.3 canonical chunk writing is available for CLI build, `string.dump`, and runtime
  round-trips.
- The official Lua 5.3.4 test archive and selected upstream fixtures are pinned; opt-in PUC Lua
  5.3 differential tests use `LUNIL_PUC_LUA53`.
- The Lua 5.3 standard-library capability matrix covers the version-specific surface currently
  implemented.
- Lua 5.3 chunk string size encoding, floating-byte table hints, numeric-for control flow,
  generic-for lowering, open `SetList`, comparison materialization, and binary concatenation are
  covered by PUC Lua 5.3 differential cases.
- A pinned PUC Lua 5.3.6 compiler chunk exercises generic-for, vararg, closure, table, comparison,
  concatenation, and Lua 5.3 numeric-string behavior during runtime tests.
- The executable Lua 5.3.4 upstream user-mode corpus now covers 23 pinned scripts, including calls,
  coroutines, metamethods, I/O, package loading, math, strings, UTF-8, packing, sorting, large RK
  constant tables, closures, goto, varargs, and scanner behavior.
- Lua 5.3-specific standard-library behavior now includes one-result `io.lines`, legacy
  `math.random` intervals and `randomseed` results, Unicode-only UTF-8 limits, legacy format/error
  text, `collectgarbage` option filtering, global `tostring` lookup in `print`, strict debug
  upvalue-index errors, and Lua 5.3 parser-stack diagnostics for long malformed input.
- Lua 5.3 label duplicate detection is scoped to the current block, while the Lua 5.4 compatibility
  path retains its enclosing-block duplicate-label diagnostic and multiline call debug-line policy.
- The build-time Lua 5.3 generator now emits the complete instruction field codec in addition to
  opcode metadata and fails compilation for a non-contiguous or non-six-bit opcode layout.
- Lua 5.3 `collectgarbage("collect")` now runs a version-profiled, resumable finalizer drain:
  finalizers run in reverse registration order, the first `__gc` failure is reported only after
  the queue is drained, and non-string finalizer errors use Lua 5.3's `error in __gc` message.
  Lua 5.4 keeps its existing warning-and-continue behavior through the generated version feature
  table.
- The original upstream `gc.lua` is now executed from the pinned Lua 5.3.4 archive, bringing the
  executable Lua 5.3 user-mode corpus to 24 scripts. Numeric-for body locals and dead-coroutine
  open upvalues are covered by the GC boundary tests.
- Version-specific behavior is now selected through generated `LuaVersionFeatures` metadata. The
  Lua 5.3 and 5.4 adapters are enabled by compile-time `LUNIL_LUA53_ADAPTER` and
  `LUNIL_LUA54_ADAPTER` capabilities (controlled by `LunilLua53AdapterEnabled` and
  `LunilLua54AdapterEnabled`), while unimplemented 5.1/5.2/5.5 identities remain explicit and
  cannot silently inherit Lua 5.4 semantics. Future adapters can be enabled through the
  corresponding MSBuild properties without duplicating the shared runtime and standard library
  paths.
