# Lunil 0.12.0-alpha.2

Lunil 0.12.0-alpha.2 introduces the first production hot-update preview for hosted and game-loop
workloads. It combines signed patch distribution, dependency-aware atomic publication, explicit
state/resource migration, multi-State rollout, and durable recovery records.

## Trusted patch bundles and tooling

- Added canonical signed Patch Bundles with ECDSA P-256/SHA-256 trust stores, payload hashing,
  expiry/channel/build/revision/ABI acceptance policy, replay checks, and bounded readers.
- Added dependency-first planning with deterministic strongly connected components and isolated
  source, binary-chunk, and host-decoded canonical-IR preflight.
- Added `lunil patch pack`, `verify`, `inspect`, `dry-run`, and `diff` workflows.

## Game-loop atomic publication

- Added background `PreparePatchAsync`, host-bound expected revisions, same-thread update windows,
  and configurable pause deadlines.
- Added rollback-capable multi-module publication for cache values, module records, table identity,
  loader upvalues, closure generations, schema versions, and JIT invalidation.
- Suspended frames retain their captured immutable function generation; calls entering after commit
  observe the replacement generation.

## State and host-resource migration

- Added signed canonical migration schemas with version compare-and-swap and RFC 6901 state paths.
- Added preserve, drop, and reversible host-adapter rules for Lua graphs and userdata payloads.
- Added coroutine, timer, event-subscription, and task dispositions with reversible host adapters
  for cancel, restart, drain, continue, and active-resource rejection policies.

## Multi-State rollout and recovery

- Added process-local all-State barrier commits, ordered canary/ring rollout, and health-gated
  rollback while update windows remain held.
- Added a canonical NDJSON deployment journal with stable-storage append flushes, contiguous
  sequence validation, SHA-256 hash chaining, transaction phase validation, and bounded readers.
- Added lifetime writer-lock ownership, concurrent verified readers, explicit retention-aware
  compaction, and opt-in automatic compaction before active-file byte or entry limits are exhausted.
- Compaction always retains incomplete transaction histories, rewrites the retained sequence and
  hash chain through a flushed same-directory temporary file, and atomically replaces the active
  journal while returning both chain-tail hashes for external anchoring.
- Added host-reconciled crash recovery records for incomplete started, prepared, or publishing
  transactions.

## Operations and limits

- Added `ActivitySource` traces and `Meter` counters/histograms under
  `Lunil.Hosting.HotUpdate` for preparation, commit, ring, rollout, rollback, and recovery outcomes.
- Added hard limits for patch modules, migration schema bytes/rules, rollout rings, and targets, in
  addition to bundle, journal, pause, and normal Lua execution limits.

## Compatibility and boundaries

- Existing `ReloadModule` behavior remains available; the new signed transactional path is opt-in.
- Candidate Lua code can cause host-visible effects outside the journal. Applications should keep
  candidate loaders deterministic and place external effects behind reversible host adapters.
- Crash recovery records deployment intent and host resolution; it does not serialize Lua heaps,
  suspended frames, CLR objects, or external resources.

See [Signed patch bundles](../docs/hot-update.md) for the runtime and CLI workflows.
