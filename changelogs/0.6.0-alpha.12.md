# Lunil 0.6.0-alpha.12

## JIT / Loop OSR

- Prepare the specialized Loop OSR emitter only after runtime exact-numeric qualification succeeds.
  Constructor, cold-function, short-loop, managed-boundary, `JIT3105`, and dynamic-code-unavailable
  paths continue without emitter initialization.
- Added `LoopOsrCompilerPrepared` so the one-time canonical emitter preparation cost is attributed
  separately from per-loop compilation. A specialized request reuses the same thread-safe preparation
  barrier before queue admission.
- Retained permanent non-exact rejection, zero negative guard churn, exact code-kind installation
  validation, explicit managed fallback, and complete `EnableLoopOsr=false` opt-out behavior.

## Performance evidence

- The real six-RID CI run `29241821560` passed exact-numeric acceptance, negative automatic rejection,
  negative startup, zero guard-failure, and zero managed-installation gates, but did not authorize
  default rollout because two ten-operation warm medians and osx-x64 first compilation p95 failed.
- Backend evidence now requires an even process count; protected CI uses six processes for an exact
  3:3 Loop OSR on/off order balance.
- Every backend throughput row now executes at least 30 warm operations. Per-RID and aggregate JSON
  record the operation count, order-balance result, preparation p95, and the existing unchanged
  startup/throughput/eligibility/guard/installation gates.
- The strengthened local six-process win-x64 record qualified with 7.547x arithmetic speedup over
  the interpreter, 119.634x over OSR-disabled execution, 0.527 ms preparation p95, 3.104 ms
  compilation p95, and negative warm medians between 0.967x and 1.019x.
- `EnableLoopOsr` remains `false` by default. A separate rollout still requires a new real six-RID
  aggregate with `AllRidsQualify=true`.

## Documentation and release

- Added ADR 0008 and updated the compiler design, performance baseline, NativeAOT guidance, bilingual
  README files, and version policy for `0.6.0-alpha.12`.
