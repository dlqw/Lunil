# Lunil 0.6.0-alpha.11

## JIT / Loop OSR

- Loop OSR natural-loop analysis no longer runs at normal function entry. It is deferred until the
  function reaches `LoopOsrBackedgeThreshold`, so cold functions, loop-free functions, and short
  negative workloads do not pay CFG/dominator/liveness or specialized-emitter initialization cost.
- The default specialized-only path now requires an observed exact-numeric runtime profile before
  queue admission. Unary and binary guard sites are sampled while the hot loop is still interpreted;
  one non-integer/non-float operand permanently rejects that loop before compilation.
- Added `AwaitingExactNumericProfile`, `NonExactNumericProfile`, and diagnostic `JIT3105`. Query APIs
  report a non-accepted pending state until runtime observation completes; rejection is exposed
  through the existing eligibility event, statistics, and query APIs.
- Non-exact metamethod loops no longer compile `GuardedExactNumericCil`, enter it repeatedly, or
  consume the guard-failure budget. `EnableLoopOsrManagedFallback=true` keeps the explicit managed
  fallback and guard-widening behavior unchanged.
- Exact-numeric qualification is completed once per loop. After acceptance or permanent rejection,
  the interpreter hot path no longer performs guard-site dictionary lookup or sampling.

## Performance evidence

- The Loop OSR on/off pair alternates order between independent benchmark processes, eliminating the
  fixed-order thermal and shared-runner bias seen in the previous six-RID evidence.
- Negative workload qualification now covers first-execution startup, warm throughput, automatic
  eligibility acceptance, managed installations, and guard failures. `lua_calls`, `table_access`,
  `metamethod`, and `coroutine_error_hook` must keep automatic acceptance and guard failures at zero.
- The balanced win-x64 five-process record (`iterations=1,000,000`, nine cold samples per process)
  reported 8.670x arithmetic speedup over the interpreter, 115.995x over OSR-disabled execution,
  3.538 ms compilation p95, zero allocation slope, 100% liveness-cache hit rate, and no negative
  workload gate failure.

## Default behavior

- `EnableLoopOsr` remains `false` by default in this release. The six-RID aggregate met code-shape,
  arithmetic, allocation, and compilation gates but did not meet shared-runner negative timing on
  four RIDs.
- `EnableLoopOsr=false` remains the complete opt-out, and `EnableLoopOsrManagedFallback=false` remains
  the release default.
