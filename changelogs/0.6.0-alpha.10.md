# 0.6.0-alpha.10

## 精确数值 Loop OSR 生产化

- 新增真实的 `GuardedExactNumericCil` Loop OSR `DynamicMethod`。verified natural loop 中的
  load/move/set-top、条件与无条件跳转、numeric-for、guarded close，以及精确整数、浮点、
  混合数值 unary/binary/comparison/bitwise 操作可直接在生成 CIL 中执行。
- OSR entry 与每次 loop header 都保留 canonical PC、instruction budget、hook/debug epoch、
  GC/finalizer、close/unwind 和寄存器 tag 守卫；guard failure、poll、预算耗尽与循环外出口均从
  已物化状态精确返回共享 scheduler，不重放副作用。
- Loop OSR 复用 owner-scoped register-liveness cache、bounded compile queue、并发去重、LRU
  code-byte budget、module invalidation、取消与 dispose 防迟到安装机制。

## 确定性资格与显式 managed fallback

- `EnableLoopOsr` 继续默认 `false`。显式开启后，只有保证生成
  `GuardedExactNumericCil` 且包含数值 hotspot 的循环才会自动进入编译队列。
- 新增 `EnableLoopOsrManagedFallback`，默认 `false`。table、upvalue、closure、vararg、call、
  tail-call 与 to-be-closed 等 managed semantic boundary 默认标记为 `Ineligible`；宿主只有显式
  开启该选项才会使用实验性的 `ManagedCanonicalProgram`。
- 新增 `LuaJitLoopOsrEligibility`、`LuaJitLoopOsrCodeKind`、
  `GetLoopOsrEligibility`/`EvaluateLoopOsrEligibility`、accepted/rejected 事件、独立统计及
  `JIT3101`–`JIT3104` 诊断。安装阶段再次验证实际 code kind，自定义 compiler 不能绕过默认门控。
- 函数首次进入时只执行一次 Loop OSR 资格分析；无循环或全部拒绝的函数永久进入 fast-rejection
  状态，后续指令与 backedge 不再重复执行 analyzer、dictionary lookup 或锁定。
- 专门化方法重复 guard failure 时，只有显式启用 managed fallback 才会扩宽到 managed loop；
  默认路径停止继续观察该循环。

## 性能证据、兼容性与 CI

- backend evidence 新增独立的 `loop_osr_off` paired baseline，并记录 OSR verification、analysis、
  liveness/cache hit、specialization planning、CIL emission、delegate creation、编译分配、code kind、
  guard/专门化数量、eligibility 与 managed installation count。
- win-x64 五进程、每进程九个 cold sample、`iterations=1,000,000` 的正式记录达到 8.516x
  interpreter arithmetic median speedup、128.737x OSR-on/off median、bootstrap 95% 下界分别为
  6.012x 与 93.992x、3.080 ms compile p95、零 allocation slope、100% liveness-cache hit 和
  44,592 B compile-allocation p95。lua-call/table/metamethod/coroutine-error-hook 的 on/off median
  分别为 0.935x、1.003x、0.992x 与 0.980x，自动 managed OSR 安装数为 0。
- 六 RID 聚合脚本新增 `loop-osr-six-rid-decision.json`，CI 同时发布 Tier 1、Tier 2 与 Loop OSR
  evidence；合成六 RID 输入已验证 code-kind、负负载、分配、liveness 与 qualification 聚合字段。
- NativeAOT 和其他动态代码不可用环境不分析、不预热也不编译 Loop OSR，并继续精确使用解释器或
  构建期 AOT。trimmed single-file、ReadyToRun、NativeAOT 与 NuGet 验证使用同一 fixture corpus。
