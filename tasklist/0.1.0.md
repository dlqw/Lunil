# 0.1.0 后续任务

本文件只记录当前里程碑明确暂缓、且已有实现路径的工作；优先级按依赖顺序排列。

## P0：运行时对象所有权与逻辑 GC

- 中文解释：当前 CLR 引用足以验证解释器语义，但不能提供 Lua 弱表、ephemeron、finalizer 顺序、对象复活和可观察的 `collectgarbage` 行为。
- 处理方案：引入 `LuaGcObject`/`LuaHeap` owner、颜色与年龄状态、根句柄、incremental/generational 步进、remembered set、弱表 fixed point 和安全点 finalizer 队列；所有 table/closure/thread/upvalue 写入统一经过 barrier。
- 依赖：稳定的 `LuaValue` 引用 tag、`LuaState` 所有权检查和表存储接口；完成后才能承诺弱表、`__gc` 与跨 state 引用拒绝。

## P0：元表、元方法与完整关闭/错误展开

- 中文解释：基线解释器目前只执行 raw table 与基础数值路径；非 nil `<close>` 值缺少 `__close` 时会正确报错，但尚未调用元方法，也未实现错误替换、链接和 protected boundary。
- 处理方案：增加 metatable/version、共享元方法查找与链长预算、`__index`/`__newindex`/算术/比较/调用/长度/拼接路径；为 frame 增加 protected boundary 和 pending error，按逆序、不可 yield 规则执行 `__close`，随后实现 `pcall`/`xpcall`。
- 依赖：逻辑 GC owner/barrier、可重入调用 continuation、Lua error value 载体。

## P0：协程、yield 与 continuation ABI

- 中文解释：栈和 PC 已显式保存，但当前 IR 没有可执行的 `Yield`，托管函数也只有同步返回 ABI。
- 处理方案：增加 `Yield`/resume signal、native continuation record、protected-call continuation 和 suspended close list；实现 create/resume/yield/wrap/close，并对 hook、`__close`、跨 C# 边界 yield 限制做差分测试。
- 依赖：完整错误展开与关闭状态机；显式 `LuaFrame`/`LuaThread` 已完成，可在现有 ABI 上扩展。

## P1：PUC prototype 与 canonical IR 双向转换

- 中文解释：PUC binary chunk 的物理模型和源码 canonical IR 都已存在，但二者之间尚未接通，因此外部 chunk 还不能由基线解释器直接执行。
- 处理方案：将全部 83 个 opcode 规范化为 canonical 操作和显式基本块；保留开放调用、`EXTRAARG` 配对、debug PC、TBC/CLOSE 状态；反向生成时做寄存器窗口压缩与 PUC 限制检查。
- 依赖：canonical IR v1 verifier、现有 `Lua54Prototype` reader/writer/verifier；元方法 slow path ABI 稳定后完成执行差分。

## P1：高性能 table storage 与字符串哈希加固

- 中文解释：当前 `LuaTable` 使用 array 加托管 `Dictionary`，适合作为语义基线，但尚不支持 tombstone 驱动的 `next` 延续、shape/version 或 runtime 私有随机 hash seed。
- 处理方案：替换 hash part 为开放寻址 bucket/tombstone，加入 rehash 预算、删除后遍历状态、metatable/shape/storage version；字符串池使用每 state 随机 seed 和可回收短字符串索引，并增加哈希洪泛基准与 fuzz。
- 依赖：逻辑 GC owner/barrier、元表版本模型；对外 `LuaTable.Get/Set` 契约保持不变。

## P1：标准库与宿主能力

- 中文解释：基线只提供注册托管函数的入口，尚未交付 Lua 5.4 标准库。
- 处理方案：先实现无外部能力的 base/table/string/math/utf8/coroutine/package 核心，再通过 `IFileSystem`、`IProcessService`、`IEnvironmentProvider`、`IClock`、`IRandomSource`、`IModuleResolver` 和 `IConsole` 实现 io/os/debug/package；逐项对照 PUC 错误条件。
- 依赖：元方法、逻辑 GC、协程、pattern/pack/unpack 子系统和 module resolver。
