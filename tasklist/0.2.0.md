# 0.2.0 后续任务

以下事项未在 0.2.0 逻辑 GC/元方法里程碑中用简化实现代替，而是按依赖关系进入后续版本。

## 1. 协程与 yield continuation

- 中文解释：当前 `LuaThread` 已保存显式栈、frame、PC、unwind 和 close continuation，但尚未公开 `coroutine.create/resume/yield/wrap/close`，托管函数也尚未区分 yieldable continuation ABI。
- 处理方案：增加 `LuaVmSignal.Yield`、resume 参数/返回区间、native continuation frame、跨 yield 的 protected/close 状态机，并用 PUC coroutine/close 差分和 GC stress 验证 suspended roots。
- 依赖：0.2.0 的 owner/根集合、protected unwind、tail-call boundary 和可恢复 `__close` 已完成；还需要先冻结 native continuation ABI。

## 2. PUC prototype 到 canonical IR 的完整执行转换

- 中文解释：reader/writer/verifier 已能读取和生成 PUC Lua 5.4 chunk，但 prototype 尚未转换成当前 canonical IR 并交给解释器执行，因此还不能运行完整官方二进制测试集。
- 处理方案：实现全部 Lua 5.4 opcode 到 canonical operation/continuation 的转换，保留 open call、TBC/CLOSE、debug PC 和 upvalue 来源；对读入与自产 chunk 分别做 PUC 5.4.8 双向差分。
- 依赖：canonical IR v2、运行时元方法、错误展开和逻辑 GC 已完成；依赖协程/yield opcode 的 continuation 表示。

## 3. 完整标准库与宿主能力层

- 中文解释：当前只安装验收所需的 `pcall`/`xpcall` intrinsic，测试中的 `setmetatable`、`emit` 等为受控宿主 fixture，尚不是可发布标准库。
- 处理方案：按 basic、coroutine、package、string、utf8、table、math、io、os、debug 顺序实现，并把文件、进程、环境、时钟、随机数和控制台放入可替换 capability provider；原生 loader 返回稳定的不支持诊断。
- 依赖：协程 ABI、chunk 执行转换、userdata/thread 类型补全和 warning/debug hook 设计。

## 4. 调用路径固定分配与解释器吞吐优化

- 中文解释：numeric-for 每迭代分配已消除，但每次解释执行仍因 frame、参数数组、call resolution 数组和最终 immutable result 产生约 5KiB 固定分配；这不随循环次数增长，但尚未达到设计中的 Lua-to-Lua 调用零参数数组目标。
- 处理方案：使用 Lua 栈窗口传递参数/结果，以 frame pool 或值型 frame storage 管理 continuation，只在宿主边界物化数组；每项优化保留 GC root map、owner 校验和 deopt 兼容测试。
- 依赖：先冻结协程/native continuation ABI和后续 CIL backend 共用的 Runtime ABI；不得绕过 barrier 或 logical roots。

## 5. 官方套件、跨平台 fuzz 与后端差分

- 中文解释：0.2.0 已有固定 PUC 5.4.8 差分、GC stress 和确定性 fuzz，但尚未声称通过完整 PUC 官方套件，也未覆盖 Windows/Linux/macOS x64/arm64 矩阵。
- 处理方案：在 chunk 执行和标准库完成后接入官方套件，增加长时间弱表/ephemeron/finalizer fuzz、AFL/SharpFuzz 输入、跨平台 benchmark，以及 interpreter/JIT/AOT 逐结果差分。
- 依赖：任务 1 至 3、CI 平台矩阵，以及后续 CIL JIT/AOT backend。
